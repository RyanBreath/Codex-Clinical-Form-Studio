# Codex Clinical Form Studio — 3-Minute Demo Script

## 0:00–0:30 — The Clinical Trial Data Problem

**Voice-over:**

Clinical trials depend on accurate, standardized, and traceable data collection.

However, transforming a clinical protocol into an electronic case report form, or eCRF, is still a highly manual process.

Clinical teams must interpret the protocol, identify the required data fields, define validation rules, and map each field to standards such as CDISC CDASH.

When the protocol changes, even a small amendment can affect multiple forms, field definitions, validation rules, and downstream data mappings.

Without a structured source of truth, it becomes difficult to identify what changed, which fields are affected, and whether the collected data can still be consistently recognized and analyzed.

**Visual:**

Protocol document → spreadsheets → manually created forms → disconnected mapping tables.

---

## 0:30–0:48 — Introducing the Solution

**Voice-over:**

We created Codex Clinical Form Studio, an AI-assisted, human-in-the-loop workflow that transforms clinical protocols into validated and deployable eCRFs.

The platform connects protocol analysis, structured field editing, CDASH mapping, browser-based quality assurance, version comparison, and continuous deployment in one traceable pipeline.

**Visual:**

Protocol → AI → YAML → Clinical Review → CDASH → JSON → HTML → QA → GitHub → Production.

---

## 0:48–1:15 — Protocol Analysis and YAML Creation

**Voice-over:**

The process begins with the clinical study protocol.

AI analyzes the protocol and extracts potential forms, fields, data types, units, required conditions, options, and validation rules.

The results are stored as YAML.

YAML becomes the human-readable and version-controlled source of truth for the clinical form specification.

Instead of directly editing application code, clinical users work with the structured specification through a web-based interface.

**Visual:**

Protocol text highlighted by AI → generated YAML → Form Builder interface.

---

## 1:15–1:48 — Clinical Editing and CDASH Mapping

**Voice-over:**

A study nurse or clinical specialist can add, edit, reorder, or remove fields directly from the web interface.

They can change a field type, define minimum and maximum values, configure required fields, or update selectable options without manually editing source code.

For each field, the platform can query an authorized CDISC data source for possible CDASH mappings.

GPT-5.6 evaluates the field definition together with the protocol context, ranks the most relevant candidates, and explains each recommendation.

The AI provides assistance, but the final mapping decision always remains with the clinical professional.

**Visual:**

Edit Weight field → search CDASH → display ranked candidates → clinical user clicks Approve.

---

## 1:48–2:08 — JSON and eCRF Generation

**Voice-over:**

After clinical review and approval, the YAML specification is converted into a standardized JSON model.

Our JSON-to-HTML rendering engine then generates the executable eCRF, including field controls, required settings, allowed values, ranges, and validation behavior.

The generated form is deployed to a validation site where clinical users can review the real user experience before release.

**Visual:**

Approved YAML → JSON preview → JSON-to-HTML engine → runnable eCRF.

---

## 2:08–2:32 — Automated Browser QA

**Voice-over:**

The platform then performs automated quality assurance using Playwright in a real browser.

The QA workflow verifies that every field and validation rule defined in the structured specification is correctly implemented.

For example, it confirms that required fields cannot be skipped, numeric limits are enforced, valid options are displayed, and invalid values are rejected.

When a test fails, Codex helps identify whether the issue comes from the specification or the rendering engine.

The problem can then be corrected, regenerated, and tested again before approval.

**Visual:**

QA test runs → intentional validation failure → Codex diagnosis → fix → all tests passed.

---

## 2:32–2:46 — Protocol Version Comparison

**Voice-over:**

When a new protocol version is introduced, the system compares the previous and current YAML specifications.

It clearly identifies added, removed, and modified fields, including changes to validation rules, required status, selectable options, and CDASH mappings.

This makes protocol amendments easier to review and trace.

**Visual:**

Version 1.0 versus Version 1.1 diff page.

---

## 2:46–3:00 — CI/CD and Production Deployment

**Voice-over:**

Once clinical review and QA are complete, the approved version is pushed to GitHub.

A CI/CD pipeline builds, tests, and deploys the validated eCRF to the production environment.

Because the output is standards-based and independently deployable, it can be integrated with different clinical trial management systems.

Codex Clinical Form Studio turns protocol changes into traceable, testable, and production-ready clinical forms—with human approval at every critical step.

**Visual:**

Git commit → GitHub Actions → successful deployment → production eCRF.
