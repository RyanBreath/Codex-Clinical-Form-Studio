# **AirwayAI Clinical Trial Copilot 系統開發規格書**

本規格書整合並規範了 AirwayAI Clinical Trial Copilot 團隊（臨床標準專家、系統分析與前端架構師、後端與 AI 設計師、QA 驗證與模擬數據專家，以及指導者/計畫主持人）的「通用樣板協作規範」與「AI 與系統確定性邏輯分工架構」。

---

## **一、 系統架構設計與核心理念**

在建立高合規要求的醫療級 Clinical Research Control Plane 時，核心理念為：
> **「AI 負責解析、生成與配置；系統負責邊界防守、邏輯運算與畫面渲染；最終由人類專家授權。」**

### **1. AI (Codex & Agent) 負責的領域：彈性、解析與生成**
AI 負責處理非結構化資料的轉換與繁瑣的代碼編寫，定位為「具備強大工具箱的工程執行者」：
* **非結構化資料解析 (Protocol Parsing)**：讀取 Word/PDF 臨床試驗計畫書，萃取臨床終點、收案條件與評估時程，轉化為機器可讀的 `program.yaml`。
* **國際標準智能對齊 (CDISC Mapping)**：根據醫學上下文，自動從 `terminology.yaml` 中挑選合適的 CDISC SDTM 變數代碼進行對齊與綁定。
* **結構與測試生成 (Artifacts Generation)**：根據上游變更，自動撰寫或更新 `crf-schema.json`、資料庫 Migration 腳本、以及邊界測試所需的模擬病人數據 (Synthetic Data)。
* **沙盒編譯與自動化 PR (Sandbox & Git Automation)**：在隔離的雲端沙盒中執行命令，修改多個關聯檔案後，自動建立 Git Pull Request、產出 Diff 報表，並在 `changeLog.md` 留下紀錄。

### **2. 系統 (Deterministic System) 負責的領域：確定性、渲染與防線**
系統邏輯負責處理必須 100% 準確、不可有幻覺的任務，作為產品真正的技術護城河：
* **動態畫面渲染 (JSON-Driven Rendering)**：前端 `FormRenderer.tsx` 不依賴 AI 撰寫 HTML。系統直接讀取 AI 產出的 `crf-schema.json`，根據預製的元件庫動態生成 eCRF 介面，達成「免重新編譯 (No Hardcoding)」且永不跑版。
* **數值與統計運算 (Deterministic Calculation)**：AI 絕對不能進行數據心算。所有的 AHI 下降率、Responder 判定、統計 P-value 都必須交由系統底層的 Python/R 程式或單元測試 (Pytest) 進行確定性運算。
* **資料庫微觀版控與相容性 (Schema Drift Control)**：系統強制解耦「填寫紀錄 (Data)」與「表單架構 (Schema)」。當版本升級 (v1.0 ➔ v2.0) 時，引擎負責加載對應歷史版本的 JSON 規格，確保舊病歷完美讀取。
* **安全閘口與邏輯攔截 (Guardrails & Human-in-the-loop)**：當偵測到 `🚨 [CROSS-AGENT IMPACT]` 標籤時，系統 CI/CD 自動鎖死合併權限，強制要求人類專家（計畫主持人）審查簽章後才能放行。

## **二、 專案資料夾結構與角色權限 (Template Directory Layout)**

此結構為通用研究專案之樣板。每個角色在自己定位的專屬目錄下擁有主要修改權限，並需遵循對應的角色規範定義。

```
airwayai-copilot-template/           # 專案根目錄 (通用樣板)
├── AGENTS.md                        # 全域 Master 規範 (定義角色邊界與跨界修訂規範)
├── changeLog.md                     # 置頂變更日誌 (記錄跨角色變更與臨床變更傳導)
│
├── agent.md                         # 【全域 Master 規範】定義專案架構、技術標準與協同溝通合約
├── agent-clinic.md                  # 【臨床標準專家規範】臨床領域與 CDISC 欄位規格專家定義
├── agent-SA.md                      # 【系統分析與前端架構師規範】JSON Schema 設計與動態渲染架構專家定義
├── agent-SD.md                      # 【後端與 AI 系統設計師規範】提示詞與後端版控 API 開發專家定義
├── agent-QA.md                      # 【QA 驗證與模擬數據專家規範】測試資料生成與跨版本相容性驗證專家定義
│
└── template/                        # 通用樣板主目錄
    ├── study-spec/                  # ─── 【臨床標準專家 專屬領地】 ───
    │   ├── program.yaml             # 臨床研究計畫書結構化規格 (例如收案與評估時間)
    │   └── terminology.yaml         # CDISC 控制詞彙與欄位標準
    ├── crf/                         # ─── 【系統分析與前端架構師 專屬領地 (UI)】 ───
    │   └── FormRenderer.tsx         # 動態表單渲染元件核心
    ├── data-dictionaries/           # ─── 【系統分析與前端架構師 專屬領地 (合約)】 ───
    │   └── crf-schema.json          # 根據研究規格編譯出的 JSON Schema
    ├── pipelines/                   # ─── 【後端與 AI 系統設計師 專屬領地】 ───
    │   ├── ai-compiler/             # AI 擷取與編譯 Protocol 之提示詞模板
    │   └── migrations/              # 資料庫 schema 遷移與版本控制腳本
    └── tests/                       # ─── 【QA 驗證與模擬數據專家 專屬領地】 ───
        ├── synthetic-data/          # 模擬病人測試數據集 (CSV/JSON)
        └── test_consistency.py      # 臨床規格、CRF、與後端資料的一致性驗證測試
```

## **三、 角色定義與專屬 Skills 組合**

為了落實「AI 配置 vs 系統渲染」的架構，各個角色配置了專屬的執行技能 (Skills)：

| 角色定位 | 專屬 Skills 與運作行為 |
| :--- | :--- |
| **🧬 臨床標準專家 (Clinic Agent)**<br>臨床規格萃取與標準化 | **Skill 1: Protocol Parser (計畫書結構析取)**<br>讀取計畫書檔案，將文字轉化為機器可讀的 `program.yaml`，找出收案條件、主要評估終點 (如 AHI) 與時程。<br>**Skill 2: CDISC Mapper (國際標準綁定)**<br>調用 `terminology.yaml`，將臨床變數自動對齊國際標準 (如判定「年齡」需加上 `domain: "DM"`)。 |
| **📐 系統分析與前端架構師 (SA Agent)**<br>元件映射與動態邏輯 | **Skill 3: UI Component Analyzer (元件映射與配置)**<br>分析臨床變數型態，從元件庫中挑選最適合的 UI 元件寫入 `crf-schema.json`（例如看到「三維座標」即指定 `field_type: "coordinate_3d"`，不自己寫 HTML）。<br>**Skill 4: Interdependence Builder (表單跳欄與依賴邏輯)**<br>分析條件邏輯並轉化為 JSON 驅動的跳欄規則 (如 `condition: "value < 20"` 則隱藏某元件)，由系統框架自動執行。 |
| **💾 後端與 AI 系統設計師 (SD Agent)**<br>後端版控與系統過渡 | **Skill 5: Prompt Templating (精準提示詞工程)**<br>撰寫與優化 LLM Prompt，確保 AI 解析計畫書時的穩定度。<br>**Skill 6: Schema Migration (資料庫遷移與版控)**<br>當 Schema 變更時，自動產生對應的資料庫 Migration 腳本，確保系統在讀取舊病歷時能用舊版 Schema 呈現。 |
| **🧪 QA 驗證與模擬數據專家 (QA Agent)**<br>邊界驗證與防線守護 | **Skill 7: Synthetic Data Generation (邊界測試資料合成)**<br>根據 Schema 限制生成「正常」與「超出邊界」的模擬病人資料，測試系統攔截能力。<br>**Skill 8: Consistency Execution (沙盒一致性測試)**<br>在隔離沙盒中執行 `test_consistency.py`，驗證 YAML、JSON 與資料庫三方是否一致，未通過則退回變更。 |

## **四、 跨角色協作與變更傳導機制 (Change Propagation)**

若因臨床規格變更（如 Protocol Amendment）需要進行跨越專屬目錄的修改時，必須遵循以下流程與申報機制：

### **1. 變更申報規則**
* **內部變更**：若變更僅限於角色自身專屬目錄，使用常規日誌格式記錄即可。
* **跨界變更 (Cross-Agent Impact)**：若修改非專屬目錄下的檔案，**必須**在日誌最頂部以 `🚨 [CROSS-AGENT IMPACT]` 標籤進行強烈高亮視覺凸顯，並標註發起角色與異動原因。

### **2. changeLog.md 跨界日誌範例**
```markdown
## [Active Draft / Pending Approval]

### 🚨 [CROSS-AGENT IMPACT] Protocol Amendment: Primary Endpoint Evaluation Extended to Week 6
* **Initiator:** `agent-clinic` (臨床標準專家 / guided by `agent-clinic.md`)
* **Date:** 2026-07-14
* **Trigger Event:** Decision to shift primary endpoint evaluation time window from Week 4 to Week 6.
* **Cross-Agent Changes Executed:**
  * 🔴 **`agent-clinic` (臨床標準專家)** modified `/template/study-spec/program.yaml` to update `duration_weeks: 6`.
  * ⚠️ **`agent-SA` (系統分析與前端架構師)** auto-triggered to update `/template/data-dictionaries/crf-schema.json` to alter visit mappings.
  * ⚠️ **`agent-SD` (後端與 AI 系統設計師)** updated database migration schema for safe transition.
  * ⚠️ **`agent-QA` (QA 驗證與模擬數據專家)** adjusted consistency test scripts to generate week 6 synthetic parameters.
* **Human Approval Status:** `[PENDING]` (Requires PI/計畫主持人 sign-off before Git merge).
```

### **3. Protocol 變更傳導接力工作流**

| 步驟 | 動作 | 負責方 |
| :--- | :--- | :--- |
| **1. 觸發變更** | 使用者將主要評估時間從 Week 4 改為 Week 6 | **人類專家 (計畫主持人/PI)** |
| **2. 規格與標準更新** | 解析新時程，重新對齊 CDISC 標準，更新 `program.yaml` | **臨床標準專家 (Skill 1, 2)** |
| **3. 介面配置修改** | 修改 `crf-schema.json`，將新欄位映射到系統現有的 UI 元件 | **系統分析與前端架構師 (Skill 3, 4)** |
| **4. 產生遷移與測試** | 生成 Database Migration 腳本，並合成最新的測試數據集 | **後端與 AI 設計師 (Skill 6) & QA 專家 (Skill 7)** |
| **5. 執行驗證與申請** | 沙盒中執行 Pytest，驗證無誤後，標記 `🚨 [CROSS-AGENT IMPACT]` 並建立 PR | **QA 驗證與模擬數據專家 (Skill 8)** |
| **6. 授權與鎖定機制** | 系統攔截 PR，不允許自動合併，等待人類專家審核並同意變更 | **確定性系統 (CI/CD)** |
| **7. 動態表單重構** | PR 合併後，畫面免重新編譯，瞬間讀取新 JSON 重構含有 Week 6 欄位的 eCRF | **確定性系統 (FormRenderer)** |

## **五、 安全防護憲法與安全閘口**

1. **上游觸發 (Trigger-based Edit)**：只有在有上游（如 `/study-spec/`）臨床規格變更為前提下，才被允許去提議修改其他角色的目錄檔案。
2. **角色對齊**：所有的程式碼變更與 Pull Request 提交時，必須讀取對應的 `agent-*.md` 規則，並在 Commit message 或 PR 說明中宣告對齊其職責定位。
3. **安全閘口**：含有 `🚨` 標籤的變更將被 CI/CD 自動掛起並鎖定，必須由人類決策者（計畫主持人/PI）手動核准後方可合併至主分支。
